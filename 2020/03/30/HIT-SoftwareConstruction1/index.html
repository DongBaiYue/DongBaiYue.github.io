Reading 1：静态检查

类型：一系列值的集合，以及这些值对应的操作。分为原始类型和对象类型(也称引用类型)。

静态类型：指类型检查发生在编译时，Java是一种静态类型的语言。在静态类型语言中，所有变量的类型在编译时就已经知道了，因此编译时可以进行类型检查。

动态类型：类型检查发生在程序运行时，例如Python。

静态检查：程序运行前进行的检查，静态类型检查是静态检查的一部分。常见的静态检查：

  • 语法错误，例如多余的标点符号或错误的关键词。
  • 错误的名字，如错误的方法名。
  • 参数个数
  • 参数类型
  • 错误的返回类型

静态检查倾向于与特定值无关的错误。

动态检查：程序运行中进行的检查。常见的动态检查：

  • 非法的变量值，如整数除”0”。
  • 无法表示的返回值。???
  • 越界访问。
  • null对象解引用。

动态检查倾向于特定值才会触发的错误。

类型修饰符：

public：程序任何位置都可以访问。

private：想同类的代码才在可以访问。

static：只会有一个实例化的对象。

final：引用不可改变。

6.031的3个目标：

  • safe from bugs(SFB)：远离bug。确保现在和将来都是正确的。
  • easy to understand(ETU)：易于理解、易读性。我们必须和将来接手的程序员交流(这也可能是几个月或几年后的你自己)。
  • ready for change(RFC)：可改动性。软件总是在更新迭代，一些好的设计可以让改动更容易。

静态检查与我们的目标的联系：

远离bug：静态检查可以在运行程序前发现类型错误等bug。

易读性：所有变量类型在代码中明确定义，便于理解。

可改动性：静态检查可以在你修改代码时定位出也需要修改的部分。如：当你修改一个变量名称时，编译器立即就会在所有使用这个变量的地方显示错误，提示你也应该修改它们。

Reading 3：测试

验证：验证的目的是发现程序中的问题，以提高你对程序正确性的信心。验证包括：

  • 形式推理：即通过理论推理证明程序的正确性。
  • 代码评审：即让别人认真的阅读、审校、评价你的代码。
  • 测试：即选择合适的输入、输出用例，通过运行程序检查程序问题。

测试应具备的心态：作为一个测试者，应想方设法让程序崩溃。

测试优先编程(Test-first Programming)

在测试优先编程中，测试程序优先于代码完成，编写函数的步骤是：

  1. 为函数写一个规格说明
  2. 为上一步写出的规格说明写一些测试用例
  3. 编写实际代码。一旦代码通过了所有测试用例，这个函数就算完成了。

规格说明：描述了函数的输入输出功能。它确定了函数参数的类型和对他们的所有约束(如sqrt函数的参数应是非负的)、返回值类型以及返回值与输入的关系。在代码中，规格说明包括函数签名和一些描述函数功能的注释。

通过分区的方法选择测试用例

测试思想：通过尽可能少的测试用例(以便快速完成测试)，验证尽可能多的情况。

分区：将输入空间划分为几个子空间，每个子空间内都是相似的数据，我们在每个子空间内选取一些数据，合并起来就是我们需要的测试用例。其原理是相似的数据在程序中的行为类似，所以我们可以用一小部分代表整体。

注意分区的边界：bug经常在各个分区之间的边界处发生，例如：

  • 在正整数和负整数之间的0
  • 数字类型(如int、double)的最大、最小值
  • 空集，如空的字符串、列表、数组
  • 集合类型中的第一个或最后一个元素

覆盖分区的两个极限情况：

  • 完全笛卡尔乘积
  • 每个分区被覆盖一次即可

实际测试中我们经常基于实际情况在两个极限中折中。

黑盒测试与白盒测试

黑盒测试：只依据函数的规格说明来选择测试用例，而不关心函数是如何实现的。

白盒测试：在考虑函数的具体实现方法的前提下选择测试用例。例如，如果在函数的实现中，对不同输入采用不同处理方式，那么应根据这些不同的输入区域进行分类。

覆盖率

覆盖率：测试对程序的测试程度。常见的覆盖率有：

  • 声明/语句覆盖率：每一个声明/语句都被测试了吗？
  • 分支覆盖率：每一个分支都被测试了吗？即每个判断取真或假至少各一次。
  • 路径覆盖率：每一种分支的组合路径都被测试了吗？即覆盖所有可能的路径。

其中，分支覆盖率比语句覆盖率严格(需要更多测试)，路径覆盖率比分支覆盖率严格。

单元测试vs集成测试和桩

单元测试：对孤立的模块进行测试。bug的范围小。

集成测试：对组合起来的模块进行测试。bug的范围大，但仍然是必要的，因为程序经常因为模块间的交互产生bug。另外，如果我们已经完成单元测试，一定程度上能够确保各个模块孤立的正确性，那么bug的范围也会小很多。

桩(stubs)：一种代替调用模块以确保测试模块孤立性的手段。

自动化测试和回归测试

自动化测试：自动地运行测试对象，输入对应的测试用例，并记录结果的测试。

回归测试：称修改代码后带来新的bug的现象为“回归”，修改代码后重新运行所有的测试为“回归测试”。

自动化回归测试：回归测试与自动化测试相结合，一个best-practice(最佳实践).

与我们的目标的联系：

  • 远离bug。测试的意义在于发现程序中的bug，而“测试优先编程”的价值在于尽可能早地发现这些bug。
  • 易读性。注意正确书写测试用例。
  • 可改动性。当我们修改程序后，自动化回归测试能够帮助我们杜绝新的bug产生。

Reading 4：代码评审

章节内容：$\begin{cases}代码评审：阅读并讨论别人写的代码\\好代码的书写原则 \end{cases}$

代码评审

代码评审的目的：$\begin{cases}提升代码质量\\提升程序员的水平\end{cases}$

风格标准：大多数公司或者大的项目都会要求代码有统一的风格标准，本章下面内容将讨论一些与我们的目标（远离bug、易读性、可改动性）息息相关的代码风格。

  • 别写重复的代码（Don’t Repeat Yourself）

别重复代码（Don’t Repeat Yourself）简称为DRY，是面向对象编程的基本原则。如果一段代码在几个不同的地方出现，那么当你修改这段代码时，你需要修改所有包含这段代码的区域，这样无形中增加了维护成本和发生bug的几率。把重复的代码抽象提取出来，这样当改动代码时只需要改动一处。

  • 仅在需要的地方注释

好的注释会使得代码易于修改，远离bug（因为一些重要的设想已经写出来了），并且减小了改动的难度。一些重要的注释包括规格说明、代码引用声明等。

需要注意的是仅在需要的地方注释。应当假定注释的阅读者是掌握这门程序语言的编程人员，若代码对阅读者而言是容易理解的，则不需要注释；反之则需要。

  • 快速报错/失败（Fail-fast）

快速报错是指代码应该尽可能快/早的将其中的bug暴露出来。因为问题暴露的越早（越接近bug所在区域），其修复工作也会越容易。静态检查 早于 动态检查 早于 产生错误结果。

  • 避免幻数

代码中1和0以外的常数被称为“幻数”，因为它们好像不知道从哪突然冒出来一样。解决幻数的一个办法就是写注释，但是另一个更好的办法是声明一个具有合理名字的常量。

  • 每个变量有且仅有一个目的

不要重利用参数，也不要重利用变量。在现在的计算机中，变量不是一个稀缺的资源。当你需要的时候就声明一个（命名一个易理解的名字），不需要它的时候就停止使用。如果你的变量在前面几行代表一个意思，在后面又代表另一个意思，你的读者会很困惑的。

特别地，方法的参数不应该被修改（这和“易改动性”相关——在未来这个方法的某一部分可能会想知道参数传进来的初始值，因此你就不应该在半路修改它）。所以应该使用final关键词修饰参数（这样Java编译器就会对它进行静态检查，防止重引用），然后在方法内部声明其他的变量使用。

  • 使用好的名称

好的方法名和变量名应当能够自我解释，具有明确的描述性，要做到这些往往需要比较长的命名。

在Java中，命名传统：

方法：驼峰命名法；methodsAreNamedWithCamelCaseLikeThis

变量：驼峰命名法；variablesAreAlsoCamelCase

常量：全大写，单词之间用_连接；CONSTANTS_ARE_IN_ALL_CAPS_WITH_UNDERSCORES

类：首字母大写，ClassesAreCapitalized

在保证含义清晰的前提下，命名应尽量简洁，但要避免使用缩写（除非这种缩写是公认的）。

  • 使用空格提高可读性

在代码行中添加一些一致的空格有助于人们的阅读。

需要注意的是，不要使用\t字符来代替空格，因为不同的工具在显示\t字符的时候长度不一样，有的是8个空格，有的是4个空格，有的是2个空格，所以不同的编辑器看同一份代码很可能就会显示的不一样。

  • 不要使用全局变量

全局变量意味着可以被程序的任何地方访问。下面列举了几个全局变量的缺点（并非全部）：

  • [ ] 任何方法都可能修改全局变量，容易发生意料之外的错误，且调试困难。
  • [ ] 容易造成命名冲突。
  • [ ] 破坏了函数的封装。
  • [ ] 难以并发。

当然，我们可以使用全局常量。全局常量可以在任何位置读取，但是永远不会改变，因此风险也就没有了。

  • 方法应该返回结果，而不是打印它

通常来说，只有最高层的代码才会处理与人/控制台的交互，其余代码并不需要打印出结果。唯一的例外是debug的时候。

与我们的目标的联系：

  • 远离bug。代码评审通过人的审查来发现bug。DRY使得你只用在一处地方修复bug，避免bug的遗漏。注释使得原作者的假设很清晰，避免了别的程序员在更改代码的时候引入新的bug。快速报错/失败使得bug能够尽早发现，避免程序造成更大的错误。避免使用全局变量使得修改bug更容易，因为局部变量只能在特定的区域修改。
  • 易读性。对于一些隐晦的bug，代码评审可能是唯一的发现方法。使用好的注释、避免幻数、变量目的单一化、选择好的命名、合理使用空格都可以提升代码的易读性。
  • 可改动性。DRY的代码更具有可改动性，因为代码只需要在一处进行更改。返回结果而不是打印它使得代码更可能被用作新的用途。

Reading 5：版本控制

本章内容：介绍git的基本使用方法。

git简介：git是一个分布式版本控制工具，它可以实现在文件不同版本间进行切换，分布式是相对于集中式而言，集中式的版本库集中存放在中央服务器上，而分布式的版本库可以同时存放在所有人的电脑上。

创建版本库

版本库需要创建在一个目录下，能够管理该目录下的文件。

创建方法：进入要创建版本库的目录，通过git init命令在该目录下创建仓库，刚创建的仓库为空。

把文件添加到版本库，分为两步：

  1. git add filename 暂存filename这个文件，即提交filename这个文件到暂存区。
  2. git commit -m "note" 提交暂存文件到版本库，提交说明为”note”，每一次提交都是一个新的版本。

版本切换/回退

版本切换需要知道要切换到哪个版本，可以有两种方式

  • git reset --hard HEAD^ 回退到上一个版本。HEAD表示当前版本，上一个版本就是HEAD^，上上一个版本就是HEAD^^，也可以写成HEAD~2。
  • git reset --hard commit ID 每一个版本都会有一个版本号(commit ID)，该命令可以切换到相应的版本。通过git log命令可以查看提交日志（包括版本号），版本号可以只写前几位，能够区分是哪个版本即可。

撤销修改

添加远程库：

  1. git remote add origin 将本地仓库与远程仓库链接，远程库的名字就是origin（git默认叫法，也可以起其他名字）
  2. git push -u origin master 把本地库master分支的内容推送到远程origin，第一次推送时加-u参数，以后就不需要了。

从远程库克隆：git clone

Reading 6：规格说明

规格说明：一份对双方都有制约的合同。规格说明的作用：

  • 对使用者而言，规格说明使得使用者不需要阅读源码，规格说明比源码简单易读。
  • 对实现者而言，规格说明给了实现者更改实现而不需要告诉使用者的自由，还可以限定一些特殊的输入。
  • 规格说明将使用者与实现者隔离开，使用者和实现者的代码可以自由改动，只需要双方都遵循规格说明中的制约。

前置条件：对使用者的制约，是方法被调用时应满足的条件。

后置条件：对实现者的制约，是方法被调用后应执行的行为。

测试与规格说明

测试必须满足前置条件。

已检查(Checked)异常和未检查(Unchecked)异常

已检查异常：允许的特殊输入，定义时声明为RuntimeException的子类。

未检查异常：意料之外的输入，定义时声明为Exception的子类。

规格说明的惯例

  • null是默认不被允许的；
  • 若spec没有说明，不应改变对象；

Reading 7：设计规格说明

规格说明可分为：

  • 操作性的：给出了实现过程的步骤
  • 声明性的：仅仅给出输出/结果与输入的关系

一般而言，声明性的规格说明会更好，其对实现者的约束宽松而且也更简洁。

规格说明的强度

我们说一个规约更强，是说它的前置条件更弱、后置条件更强，这两个“更弱”、“更强”中可以有一个是“等于”。显然，任何时候规约更强的实现方法都可以替换规约更弱的实现方法。

Reading 8：可变性与不变性

可变性与规约

  • 使规约更复杂。对可变对象，它在程序中可能有多个引用（或称别名），而每一个引用都能够改变它。因此在规约中，为了保证可变对象的某些性质，需要约束可变对象的每一个引用。
  • 降低了可改动性，一些改动可能带来隐秘的bug。

可变对象相比于不可变类型会有性能上的优势，但也产生了很多风险。

Reading 9：避免调试

在这篇阅读中，我们介绍了几种最小化调试代价的方法：

  • 避免调试
    • 使用静态类型检查、动态检查、不可变类型和不可变索引让bug无法产生。
  • 限制bug范围
    • 通过断言检查、快速失败让bug的影响不扩散。
    • 通过增量式开发和单元测试让bug尽量只存在于刚刚修改的代码中。
    • 最小化变量作用域使得搜寻范围减小。

增量式开发：每次只完成一小部分的开发，然后对这部分进行完全的测试，随后再进行下一小部分的开发。这可以讲大多数bug控制在我们刚刚修改/增加的代码中，以降低debug的困难。

最小化作用域：

  • 永远在for语句内部声明循环量
  • 尽量在使用变量的时候才声明它，并尽量将它放在最内部的括号内
  • 尽量避免使用全局变量

Reading 10：抽象数据类型

Reading 15：相等

相等可分为观察相等和行为相等，

  • 观察相等：两个对象在任何相同的创建者、生产者、观察者（改造者除外）操作下效果相同。
  • 行为相等：两个对象在任何相同的创建者、生产者、观察者、改造者操作下效果相同。

对于不可变对象，没有改造者，因此观察相等与行为相等等价，即值相等。

对于可变对象，观察相等指两个对象的值相等，行为相等则指对象引用相同，即指向同一个对象。

对于自定义的对象类型，默认继承自object对象的equals和hashcode方法是基于引用的。

  • 对于自定义的不可变对象类型，要实现其“值相等”的比较，就需要重写equals方法。需要保证相等的不可变对象其hashcode值一定相等，不相等的的不可变对象其hashcode值尽量不等。因为HashSet、HashMap等数据结构会按照hashcode值计算对象存储的桶，前者不符合会出现bug，后者不符合则会影响HashSet、HashMap等数据结构的性能。
  • 对于自定义的可变对象类型，若其是实现为“值相等”的比较，对象改变后其hashcode值也会发生改变，那么在HashSet、HashMap等数据结构contains等操作会出现bug（比如contains会去新的hashcode值对应的桶中寻找，结果查找失败），因此应当实现为“引用相等”，即默认继承。
